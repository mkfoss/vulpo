Conditional Compilation Switches

C/C++ supports conditional compilation by use of the following language construct:

#ifdef SWITCH_NAME

 

   // Some code

 

#else

 

   // Some other code

 

#endif

CodeBase takes advantage of conditional compilation to help support different compilers and configurations. Following are the conditional compilation switches which can be used when compiling CodeBase source code and applications. The switches are located in the 'd4all.h' header file.

If any of the switches listed in this chapter are used when compiling the CodeBase library, then they must also be used when compiling the application. The switches have been broken into several categories: compiler, file format, screen output, operating environment, client-server configuration, general configuration and spoken language.

 Note

As documented in the Getting Started manual, we suggest defining the appropriate switches at the top of include "d4all.h". It is also recommended that you run the example program d4verify.c or w4verify.c. These examples have special logic which verifies that the switches used when the library was built are compatible with the switches used when building this example.

 

Compiler Switches

CodeBase has conditional compilation switches for a number of specific compiler products. Usually, the compiler has a predefined switch that CodeBase uses. In this case, you do not have to explicitly set any switch. If you do, the name of the switch is documented in the compiler support file 'COMPILER.TXT'. Refer to the Getting Started manual.

File Format Switches

CodeBase supports several different file formats. Note that these switches change the multi-user protocol as well. This is because CodeBase is multi-user compatible with dBASE IV, FoxPro and Clipper. Only one of the following file format switches may be defined at a time.

 

Switch Name

 Description

S4CLIPPER

This switch adds support for Clipper file formats. Specifically, the .NTX index files and the Clipper memo files are supported.

When the S4CLIPPER switch is defined, all of the CodeBase functions can be used. In addition, the functions t4open and t4close can be used. Refer to the section on Clipper support in the User's Guide.

S4FOX

This switch adds support for FoxPro file formats including Visual FoxPro which supports code pages and collation sequences. This includes .CDX compound index files, compact .IDX index files, and .FPT memo files.

 

Because compound and compact index file formats are almost identical, and can easily be distinguished from one another, both are supported at the same time. When using the S4FOX switch, a single program could open up both .CDX and "compact" .IDX index files at the same time. However, the default file name extension is .CDX. If you wish to use an .IDX index file, it is necessary to explicitly specify the .IDX file name extension.

See Also: FoxPro Configuration Switches

S4MDX

This switch adds support for dBASE IV file formats.

Specifically, the .MDX index files and the dBASE IV memo files are supported.

 

FoxPro Configuration Switches

These switches provide support for the code pages and collation sequences that are used by Visual FoxPro. These switches can only be defined if the S4FOX switch has also been defined.

The following switches provide support for code pages. One or both may be defined at a time.

Switch Name

Description

S4CODEPAGE_437

U.S. MS-DOS code page number 437.

S4CODEPAGE_1251

Windows ANSI code page number 1252.

The following switches provide support for international languages.

Switch Name

Description

S4GENERAL

Adds support for the general collation sequence. This sort ordering supports English, French, German, Modern Spanish, Portuguese, and other Western European languages.

S4USE_GENERAL_TAGS_IN_RELATE

 

If this switch is defined, the CodeBase Relation/Query module will use any available tag to enable Query Optimization. This behavior can also be achieved by setting CODE4.useGeneralTagsInRelate. If this switch is not defined, only tags that use Machine collation will be used to enable Query Optimization.

 

Screen Output Switches

CodeBase can be configured to work in almost any operating environment, since it is a database management library. By default, when CodeBase needs to display output, it sends error messages to stderr and other output to stdout.

In order to configure CodeBase to output using a different output method from one of the methods described here, you need to modify the error message reporting. Refer to conditional compilation switch E4HOOK. In addition, if the report module is used, you need to refer to the report module documentation.

Define one of these switches when non-standard output is required.

Only one of these switches can be defined at once.

Switch Name

 Description

S4CODE_SCREENS

Define this switch when Sequiter's DOS screen management library CodeScreens is being used.

S4CONSOLE

S4CONSOLE should be defined when you want to run an application from the console or send output to the console.

 

Operating System Switches

The standard version of CodeBase supports DOS, OS/2 and Microsoft Windows.

If you are using Unix or a different operating environment, you need to purchase the portability version of CodeBase. The portability version comes with additional documentation and software which is useful in order to get CodeBase working under other operating systems.

There are two sets of switches in this category. The first set of switches define the type of library that will be used, static or dynamically linked. The second set of switches defines the type of operating system being used. Only one switch from each set may be defined at once.

 

Switch Name

 Description

S4STATIC

This switch must be defined when using a CodeBase static library.

S4DLL

This switch must be defined when using a CodeBase dynamic link library.

 

Switch Name

Description

S4DOS

Use this switch when compiling under DOS.

S4OS2

This switch is used when compiling under OS/2.

S4WIN16

Use this switch when compiling CodeBase under WIN16.

S4WIN32

Use this switch when compiling CodeBase under WIN32.

 

Client-Server Configuration Switches

These switches determine whether the target being built is a client or a non-client. Only one of these switches can be defined.

 

Switch Name

Description

S4CLIENT

When the S4CLIENT switch is defined, the application is a client in a client-server configuration. Therefore, when this switch is defined, the S4STAND_ALONE switch can NOT be defined.

S4STAND_ALONE

When the S4STAND_ALONE switch is defined, the application is NOT a client in a client-server configuration and the S4CLIENT switch can NOT be defined.

 

General Configuration Switches

Any number of the following switches can be defined at the same time.

 

Switch Name

 Description

S4CB51

When this switch is defined, the 5.1 API can be used in CodeBase 6. The new locking protocol that is used in CodeBase 6 will not available when this switch is defined. The following CodeBase 6 functions will not be available when this switch is defined: d4seekNext, d4seekNextDouble, d4seekNextN.

S4LOCK_HOOK

This switch modifies the CodeBase locking functions so that they call function code4lockHook when a lock fails. It is up to the programmer to define function code4lockHook. code4lockHook is designed to allow the end user the opportunity to retry the lock.

 

The function is found in the file 'C4HOOK.C' and is described by the following:

                Usage:   int code4lockHook(  CODE4 *cb,

                                                            const char *fileName,

                                                            const char *userId,

                                                            const char *networkId,

                                                            long item,

                                                            int numAttempts)

       Parameters:

                      cb   A reference to the CODE4 structure.

             fileName   This is the name of the file for which the lock contention occurred.

 

                 userId   This is the name of the user who holds the contentious lock. Use null if the userId is unknown.

            networkId   This is the network identification of the user who hold the contentious lock. Use null if the networkId is unknown.

                   item   This is the item for which the lock is contentious. The value of item can be one of the following:

                              > 0   This the record number of the locked record.

                                 0   This indicates that the append bytes are locked.

                               - 1   This indicates that the file is locked.

                               - 2   This indicates that item locked cannot be identified.

       numAttempts   This is the number of times the lock has been attempted.

 

                  Returns:

                        0   This return indicates to the top level CodeBase function to try again.

              r4locked   This return indicates to the top level CodeBase function to return r4locked.

                     < 0   This return indicates to the top level CodeBase function to return a negative value. Generally, if this is done, it is also appropriate for the code4lockHook to call error4set to set a CodeBase error.

 

NOTE: In general, calling CodeBase functions from within the code4lockHook function is discouraged since it can cause conflicts within the internal structure settings. For example, calling d4go, to change the current record in the data file for which the lock has failed, will cause internal consistency problems when returning to the function that called the hook function. However, it is acceptable to call error4set from code4lockHook, when applicable.

S4MAX

This switch adds code to limit the amount of memory functions u4alloc, u4allocErr and u4allocFree will allocate in total.

When this switch is used, it is necessary to also use the E4MISC switch.

CodeBase puts the current amount of memory allocated in the global variable long mem4allocated. The maximum amount of memory is placed in the global variable long mem4maxMemory. This is useful for simulating conditions when little memory is available. By default, mem4maxMemory is 16K. When using switch S4MAX, CodeBase assumes that mem4maxMemory is less than the actual maximum available memory.

S4SAFE

Setting this switch causes CodeBase to immediately update file lengths after information has been written to file. This avoids delayed "out of disk space" error messages when memory write-optimization is being used.

S4TIMEOUT_HOOK

When this switch is defined, CodeBase calls code4timeoutHook whenever a CodeBase client is waiting to receive a message from the server and the CODE4.timeout has expired.

It is up to the programmer to define the function code4timeoutHook. This function is designed so that the programmer may specify what action should take place when a time out occurs. This function is especially useful when used in conjunction with code4lockHook in order to determine whether CodeBase is delaying because of a lock or is waiting for a communication reply. The function is provided in the file 'C4HOOK.C'. The description of code4timeoutHook is as follows:

                     Usage:    int code4timeoutHook( CODE4 *cb, int numAttempts, long numHundredths)

          Parameters:

                             cb    This is a reference to the CODE4 structure.

         numAttempts    This the number of times the hook function has been called for this particular time out.

    numHundredths    This is the total elapsed time that the network has been delaying, in hundredths of seconds.

 

                  Returns:    A return of zero indicates that the application should continue to wait for a response. Any other value will be passed up to the calling functions.

S4OFF_THREAD 

By default, the CodeBase 32-bit library is built with some multi-threading enabled for file read/write optimization. To prevent CodeBase from using multi-threading, define S4OFF_THREAD and rebuild the CodeBase library.

This switch only controls how the CodeBase behaves internally. Defining it does not prevent the programmer from using CodeBase in a multi-threaded environment.

 

Error Switches

The following switches, when defined within the CodeBase library, determine what actions occur when an error is generated. Any number of the following switches can be defined at the same time.

Switch Name

Description

E4ANALYZE

This switch is used to perform structure analysis on internal CodeBase structures at runtime. This switch is especially useful during the development stage of applications. It should not be used during the final application compiles, since it adds significant code to CodeBase and decreases performance.

When this switch is defined, it automatically defines E4LINK and E4PARM_LOW. E4PARM_LOW checks the parameters in low level CodeBase functions.

E4DEBUG

This switch automatically defines the following error switches:

E4PARM_HIGH, E4ANALYZE, E4MISC and E4STOP_CRITICAL

E4HOOK

This switch modifies the CodeBase error functions so that they call function error4hook to display any error messages. It is up to the programmer to modify function error4hook. The prototype of error4hook as found in 'd4data.h' and is as follows:

                     Usage:    void error4hook( CODE4 *codebase, int errCode1, long errCode2, const char *desc1, const char *desc2, const char *desc3 )

 

           Description:    The purpose of the this function is to allow the programmer to easily alter or turn off error reporting. When CodeBase is built with the conditional compilation switch E4HOOK, error4hook is called by error4 and error4describe to display messages. By default, error4hook does nothing.

 

Consequently, to turn off error reporting, just define E4HOOK and do nothing else.

 

To alter error reporting, alter error4hook to display error messages as desired. Function error4hook is defined statically in 'C4HOOK.C'.

 

          Parameters:    The parameters to error4hook correspond to parameters passed to error4 and error4describe.

If error4hook is displaying error messages for error4 then the parameters desc2 and desc3 are null. The parameter desc1 may also be null depending on whether the error code has any auxiliary information associated with it.

 

Call error4text to get the error string associated with either errCode1 or errCode2. Note that if E4OFF_STRING or E4OFF is defined, the error4text function will return the string "Invalid or unknown Error Code".

                See Also:   error4, error4describe, E4OFF, E4OFF_STRING, error4text

E4LINK

This switch provides link-list error checking in the CodeBase library. Whenever an item is added or removed from a CodeBase linked list the list is "run" to verify its internal integrity. Since CodeBase linked lists are all circular, this is easy to do.

This is most useful when debugging applications which make use of the provided link-list functions. Compiling with this switch may increase the application execution time by a factor of 2 or more.

E4MISC

Using this switch adds extra code for CodeBase self diagnostics and additional error checking. Following are the effects of using the E4MISC switch:

Whenever a memory item is allocated, the returned pointer is saved in a list. When the pointer is subsequently freed, it is looked up in the list to verify that it was previously allocated. In addition, an extra 22 bytes are allocated as follows:

FFFFFFFFFFLL, Allocated Memory, FFFFFFFFFF

The 'F' represents a special fill character, and the 'L' is the number of allocated bytes. When the allocated memory is subsequently freed, a check is made to ensure that the fill characters have not changed. This is useful because C programmers sometimes modify more memory than they allocate.

Function mem4freeCheck(int maxAlloc) is also defined.

 

This function, whose prototype is defined in 'd4declar.h', returns the number of memory allocations for which there have been no subsequent free. If this number is greater than the value of parameter maxAlloc, a severe error is generated.

E4OFF

This switch disables all CodeBase error messages. This performs the same function as runtime flag CODE4.errOff. This switch does not affect the functionality of an application, however, only error numbers are available in an application. Using this switch results in a smaller executable size, since the text for the error messages is not included. Error hooks can still be used and this switch also does not effect the actual generation of CodeBase errors. When E4OFF is defined then E4OFF_STRING is implicitly defined. Defining this switch will cause error4text to return the string "Invalid or Unknown Error Code" whenever it is called.

E4OFF_STRING

This switch removes all the extended error strings from the CodeBase application. Consequently, CodeBase will display a unique error number and the basic error description associated with the error. Defining this switch will decrease the size of the application, since the long static strings will not be included in the executable. This default implementation under DOS is to have the library compiled with this switch. Defining this switch will cause error4text to return the string "Invalid or Unknown Error Code" whenever it is called.

E4PARM_HIGH

This switch causes CodeBase to include parameter checks for high-level functions. In general the library should be compiled with this switch defined. This switch should also be defined in end-user applications, since it will prevent applications from crashing when bad parameters are passed to CodeBase functions. The default setting for this switch is to be turned on.

E4PAUSE

Compiling with this switch causes CodeBase programs to prompt the user to "press a key to continue" (when S4CONSOLE is defined) whenever an error is encountered. This switch only applies to S4CONSOLE applications. The default setting for this switch is to be turned on.

E4STOP

This switch causes CodeBase to halt the execution whenever any CodeBase error is encountered. Note that using an error hook will override this switch.

E4STOP_CRITICAL

This switch causes CodeBase programs to halt whenever any critical error is encountered. Critical errors include the following:

         Out of memory.

         Corrupt memory detected.

         Corrupt structures detected.

         Unexpected internal results (e.g. corrupt index)

Using an error hook will override this switch. The default setting for this switch is to be turned on.

E4VBASIC

This switch adds extra code which validates the pointers passed to most functions. With this switch defined, an error is generated at run-time when an incorrect pointer is passed to a function. For example, passing a DATA4 pointer to a function like f4name(), which takes a FIELD4 pointer, will generate an error. This switch is only useful when the DLL is used by applications built by Visual Basic; compilers for other languages normally generate a warning or error if pointers are used incorrectly.

This switch only has an effect when S4DLL is defined.

 

 

 

 

Library Reducing Switches

The following switches, when defined within the CodeBase library, reduce the size of the library by removing some its functionality. As a result, applications linking into a reduced library will be smaller and in many cases faster than the standard library. When these switches are defined, many CodeBase functions will be unsupported. For example, attempting to append a new record to a data file with S4OFF_WRITE defined is unsupported and will generate an error. Listed below are the switches, and the functions they effect.

Any number of the following switches can be defined at the same time.

Switch Name

 Description

S4OFF_INDEX

This switch is used to build a library that does not use indexes. All code related to indexes is removed from the library. This dramatically reduces the size of the CodeBase library. The following functions are disabled and always return 'success' when used with the S4OFF_INDEX switch: d4reindex, d4tagSync. i4fileName, t4alias, t4expr, t4filter return null when this switch is defined. The following functions return uninitialized objects: d4index, d4tagSelect, d4tagDefault. When S4OFF_INDEX is defined, the following functions are unsupported and always generate an e4notIndex error if called: d4freeBlocks, d4seek, i4close, i4create, i4open, i4reindex, i4tag, t4open.

This switch can only be defined when the application has been compiled with the S4STAND_ALONE switch defined.

S4OFF_MEMO

References to memo files are taken from the library. This reduces the executable size when memo files are not needed. The following functions are disabled and always return 'success' when used with the S4OFF_MEMO switch: f4memoFree. When the memo field functions are used to access memo fields (as opposed to any other 'generic' type of field) , the following functions are unsupported and always generate an e4notMemo error if used. f4memoAssign, f4memoNcpy, f4memoLen, f4memoPtr, f4memoStr, d4memoCompress.

 

This switch can only be defined when the application has been compiled with the S4STAND_ALONE switch defined.

S4OFF_MULTI

This switch compiles a single-user version of the CodeBase library. No file locking is performed or allowed when this switch is used. If an application is to be used in a single-user environment, this switch results in smaller code sizes and much faster execution times. This switch should also be used when using a compiler that does not perform file locking. This switch should not be used when an application is running in a multi-tasking environment and more than one instance may be running. The following functions do nothing but return 'success' when S4OFF_MULTI is defined (Note: an application will be slightly smaller if they are not called): d4lock, d4lockAll, d4lockAppend, d4lockFile, d4lockAdd, d4lockAddAll, d4lockAddAppend, d4lockAddFile, d4unlock, file4lock, file4refresh, file4unlock, relate4lockAdd, code4lock, code4unlock.

This switch can only be defined when the application has been compiled with the S4STAND_ALONE switch defined.

S4OFF_OPTIMIZE

This switch disables CodeBase memory optimization. This results in slower execution time in both single and multi-user applications. It can be used to reduce CodeBase memory requirements and code size. When this switch is used, the following functions do nothing but return 'success' (Note: an application will be slightly smaller if they are not called): code4optAll, code4optStart, code4optSuspend, d4optimize, d4optimizeWrite, file4optimize, file4optimizeWrite, file4refresh.

When the switch S4CLIENT is defined, then this switch is automatically defined as the default setting.

S4OFF_REPORT

This switch can be used to remove all CodeReporter reporting functions and functionality from the CodeBase library (i.e. functions which are documented in the CodeReporter reference manual). Using this switch will reduce the size of the library and resultant executables, even for programs which do not otherwise call the CodeReporter reporting functions.

S4OFF_TRAN

Use this switch to remove transactional capabilities from the CodeBase library. This means that the transactional functions can not be called by the client, but the CodeBase database server will retain transactional file capabilities and compatibilities. For the stand-alone version of CodeBase, compiling with this switch removes the computability with other CodeBase applications that have transactional capabilities.

S4OFF_WRITE 

This switch is used to create read-only applications. All code that performs disk writes is excluded from the CodeBase library. The following functions do nothing but return 'success' when S4OFF_WRITE is defined: code4flushFiles, d4flush, d4optimizeWrite. The following functions are completely unsupported, and always generate an e4notWrite error. d4append, d4appendBlank, d4appendStart, d4changed, d4create, d4deleteRec, d4memoCompress, d4pack, d4recall, d4reindex, d4write, d4zap, f4assign, f4assignDouble, f4assignLong, i4create, i4reindex, i4tagAdd.

This switch can only be defined when the application has been compiled with the S4STAND_ALONE switch defined.

 

Spoken Language Switches

Following are the switches for spoken language support. There are two supported sort orderings for English and German.

Switch Name

Description

S4ANSI

This switch is defined to make CodeBase use the ANSI sort ordering instead of the default ASCII ordering. The ANSI sort ordering makes CodeBase use the same ordering as the Microsoft Windows comparison functions. Unfortunately, this makes CodeBase incompatible with the dBASE, FoxPro and Clipper sort orderings.

S4DICTIONARY

For the German version of CodeBase (if S4GERMAN is defined), this switch determines whether the sort order is by the default (phone book sort ordering), or dictionary ordering (if S4DICTIONARY is defined).

S4FINNISH

This switch is defined to get Finnish language support.

S4FRENCH

This switch is defined to get French language support.

S4GERMAN

This switch is defined to get German language support.

S4NORWEGIAN

This switch is defined to get Norwegian language support.

S4SCANDINAVIAN

This switch is defined to get Scandinavian language support.

S4SWEDISH

This switch is defined to get Swedish language support.
